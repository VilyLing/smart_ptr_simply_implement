## 💡 核心设计思路记录

### 思路 1: 静态 OOP 绑定
最初的尝试是让 `control_block` 强绑定 `T` 类型。该版本是从size_t* count优化而来，也是最初的base.
- **优点**：逻辑直观，符合初级面向对象思想。
- **缺点**：侵入性强，无法管理基本类型（如 `int`），且难以处理协变。

### 思路 2: 类型擦除与多态 (Current)
受 C 风格 `void*` 和多态的启发，引入了 `control_block_super`。
- **架构**：`Shared_Pointer` 持有 `super*`，子类 `final` 负责具体的 `delete` 逻辑。
- **突破**：成功实现了运行期绑定删除器，解耦了智能指针接口与底层存储细节。
- **痛点**：目前 `weak_ptr` 在处理“对象销毁但控制块保留”的逻辑上仍有 Bug。

## 🧠 关于放弃 CRTP 优化的焦虑与反思

在尝试引入 **CRTP (Curiously Recurring Template Pattern)** 来消除虚函数开销时，我遇到了严重的架构冲突：
1. **类型擦除与静态多态的矛盾**：基类需要抹除类型，而 CRTP 需要感知子类类型。方案是3冲继承导致了虚函数开销也无法解决，或者是继承模板类。
2. **学习成本与收益不成正比**：在模板推断（第 16 章）尚未完全掌握时，强行冲击库级别的优化导致了严重的学习焦虑。

**结论：** 与其在原地追求极致的性能，不如带着目前的遗憾继续向后阅读。C++ 的知识是环环相扣的，或许在学完“可变参数模板”和“完美转发”后，答案会自然浮现。
